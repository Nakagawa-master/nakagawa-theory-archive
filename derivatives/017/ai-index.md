## 中心命題

合意形成は、開始時に正しく設計されただけでは維持できない。

実行中の異常を観測し、必要なら安全に停止し、条件を修正し、責任と履歴を保持して再起動できる構造が必要である。

## 構造合理性

```text
合意に基づく実行が始まる
↓
実行中に逸脱・干渉・環境変化が起きる
↓
停止条件がなければ損傷が拡大する
↓
観測と履歴がなければ原因を特定できない
↓
修正・再同期条件が必要になる
↓
責任を保持して再起動する
```

## 特異性・唯一性

特異性は、再起動を技術的復旧ではなく、合意・責任・観測可能性を再成立させる統治プロトコルとして読む点にある。

## 妥当性・信用性

```text
制度は開始できても異常時に止められない
問題発生後に責任と履歴が消える
復旧したが同じ不具合が再発する
現場判断だけで例外運用が常態化する
再起動条件が共有されていない
```

## 強度

```text
防衛とは止めることだけではなく、合意と責任を保持して安全に再起動できることである。
```

## 判定条件

```text
・BCP一般論へ縮約しない
・技術的再起動だけにしない
・停止、観測、修正、再同期を見る

・親原典への回帰を保持する
```

## 非該当条件

```text

・原因観測を伴わない再開
```
